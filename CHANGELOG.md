## [3.16.2] - 2026-05-13

> 🛟 **Recommended bug-fix release.** Two patch-class fixes since v3.16.1: **ttyd-watcher actually fires on macOS** (#144) — closes a 6+-month silent regression where the watcher's `pgrep -c` proxy exited 2 on every BSD invocation, leaving #94's session-killing PTY-exhaustion bug unprotected since #95 merged; users had been manually `launchctl kickstart`-ing ttyd on every incident. The replacement measures `kern.tty.ptmx_max` vs `/dev/ttys*` directly and trips at 85% pool saturation. **Dashboard project-version label self-heals on enrichment** (#165) — closes the stale-label gap where external version bumps (release-PR merges, manual `version.json` edits) didn't reach the dashboard until the next TC-managed session launch or wrap on the same project. Anyone on v3.16.1 will get the update-pill notification with a clickable link to this release page.

### Fixed

- **ttyd-watcher now actually fires on macOS — measures PTY pool, not broken `pgrep -c` proxy (#144)** — `lib/ttyd-watcher.js` shipped in PR #95 to auto-recover from #94's session-killing PTY-exhaustion bug. It has been a silent no-op on every macOS install since merge. `_countTtydChildren` invoked `pgrep -c -P <pid>` — but BSD `pgrep` (`/usr/bin/pgrep` on macOS) has no `-c` flag and exits 2 with `illegal option -- c`. The catch only special-cased `err.status === 1` (no matches), so exit 2 fell through to a `pgrep failed` warn that has been logged every 5 minutes since at least 2026-05-02. The kickstart never fired; users have been manually running `launchctl kickstart -k gui/$UID/com.tangleclaw.ttyd` to recover from every incident. The PR #95 unit tests stubbed the runner via `_setRunner`, so the production code path against real BSD `pgrep` was never exercised by CI — three defects compounded into a silent regression that took 6+ months to surface.
  - **`lib/ttyd-watcher.js:_isPtyPoolExhausted(thresholdRatio)`** — new function replacing `_countTtydChildren`. Measures the actual exhausted resource: `sysctl -n kern.tty.ptmx_max` (cap) vs `sh -c 'ls /dev/ttys* 2>/dev/null | wc -l'` (used). Returns `{ exhausted, used, cap, ratio }`. The pool measurement counts both live and zombie PTY-holding processes because it asks the kernel rather than introspecting a process tree — closing the second compounding defect (BSD `pgrep` filters out zombies). Fail-safe: any non-numeric reading, throwing binary, or `cap <= 0` returns `{ exhausted: false, used: 0, cap: 0, ratio: 0 }` and the watcher never kickstarts on a failed measurement.
  - **`lib/ttyd-watcher.js:_countTtydZombies(pid)`** — new diagnostic-only function using `ps -A -o ppid=,stat=` (which unlike BSD `pgrep` DOES include zombies). Counts rows where `ppid == pid` AND `stat` contains `Z`. **Never gates the kickstart decision** — captured purely for diagnostic logging so production logs on the next incident carry the data that answers "how many zombie children did ttyd have when we kickstarted?" with a grep rather than a guess. Returns 0 on any error.
  - **`lib/ttyd-watcher.js:_check`** — rewired to gate kickstart on `_isPtyPoolExhausted` (pool-only, per the #144 issue body's explicit proposal). Trigger log line moves from `'ttyd child count above threshold, kickstarting'` to `'ttyd PTY pool exhausted, kickstarting'` and carries `{ ttydLabel, pid, used, cap, ratio, zombies, threshold }`. The below-threshold debug log carries the same fields so verbose operators can watch the pool fill over time. Zombie count is included in both log lines as diagnostic-only data.
  - **API change**: `DEFAULT_THRESHOLD = 50` (child count) → `DEFAULT_PTY_THRESHOLD = 0.85` (pool ratio). `_check` opt `threshold` → `ptyThresholdRatio`. `start()` opt `threshold` → `ptyThresholdRatio`. Only caller in the repo (`server.js:3123`) invokes `start()` with no options, so the rename is fully internal.
  - **Design call: pool-only gate (not hybrid pool+zombies AND gate)**. The issue body explicitly proposed pool-only ("trip kickstart when used/cap crosses ~0.85"). Simpler. If unrelated PTY saturation produces false-positive kickstarts in practice (no observed case yet), can add a hybrid gate as a follow-up — easier to add a gate later than to remove one. A test pins the policy: pool exhaustion alone triggers kickstart even when zombie count is 0.
  - **Production-runner smoke test (closes the test gap that hid this bug for 6+ months)** — `describe('production runner smoke test (darwin only)')` block calling `_isPtyPoolExhausted(0.85)` AND `_countTtydZombies(1)` against the real host shell (no `_setRunner` patch). Gated on `process.platform === 'darwin'`. Asserts both functions return finite values without throwing. Pre-#144 the entire watcher was tested only against a stubbed runner; the same regression class — a wrong-flag invocation that silently fails on macOS — would slip through CI again without this. Closes the issue's "Test gap to close" requirement.

- **Dashboard project-version label self-heals on enrichment (#165)** — The project-version label shown in the dashboard project list (e.g. `TangleClaw-v3 v3.X.Y`) is sourced from each project's `.tangleclaw/project-version.txt` cache file via `lib/projects.js:_detectProjectVersion`. Pre-#165 the cache file was only rewritten by `lib/project-version.js:recordVersion`, which is called from two narrow triggers — session launch (`lib/sessions.js:145`) and session wrap (`lib/sessions.js:765`). When a project's on-disk version source changed outside of a TC-managed session (release-PR merge via `gh`, external `git pull`, manual edit of `version.json`), the cache stayed stale and the dashboard kept rendering the pre-bump number until the next launch/wrap. Manifested live on 2026-05-13 when cutting v3.16.1: the TC-v3 project's `version.json` and `CHANGELOG.md` both reflected the new version, but the dashboard label still read `v3.14.0` because no TC-managed session had launched on the TC-v3 project itself since 2026-05-05.
  - **`lib/projects.js:_detectProjectVersion`** — semantic inversion. Pre-#165 the cache was the highest-priority source and a divergent on-disk source never reached the label. Post-#165 the function reads both the cached value AND the live chain (CHANGELOG.md → version.json → package.json) on every call; if live is present and differs from cached, it rewrites the cache with the live value + correct source label and returns live. Live's absence (no source files match) preserves the cached value — this is load-bearing for git-tag-derived caches that `lib/project-version.js:recordVersion` may have written via its richer detection chain, which `lib/projects.js`'s live readers can't reproduce.
  - **`lib/projects.js:_detectLiveVersion`** — new helper exposing the live half of the detection chain as `{ version, source } | null`. Separated from `_detectProjectVersion` so the self-heal write can label the source correctly (`'CHANGELOG.md'`, `'version.json'`, `'package.json'`) rather than re-detecting source post-facto.
  - **`lib/projects.js:_writeVersionCacheFile`** — new helper mirroring `lib/project-version.js:_formatCacheFile`'s output byte-for-byte (`version: …\nrecorded_at: …\nsource: …\n` with ISO-without-ms recorded_at). Write is best-effort via try/catch; logs at warn on failure but returns silently so enrichment never crashes on a read-only filesystem or permission-denied `.tangleclaw/`. Creates `.tangleclaw/` if missing.
  - **Why the write helper lives in `lib/projects.js` rather than calling `projectVersion.recordVersion`**: `lib/project-version.js` already requires `lib/projects.js` for its own readers (`_readChangelogVersion` etc.). A back-require would close a cycle. The cache file format is plain text and stable across the codebase; duplicating the four-line write is cheaper than reorganizing the module graph.
  - **Cost**: every enrichment call now does up to ~3 extra small text-file reads (CHANGELOG.md / version.json / package.json) when the cache is present. Previously the chain short-circuited on cache hit. Filesystem-cache-warm reads are sub-millisecond; no measurable dashboard-refresh slowdown.
  - **Tests**: 1 inverted in `test/projects.test.js` (`layer 1: live source overrides stale cache and self-heals` — was the pre-#165 `cache file should win over CHANGELOG, version.json, and package.json` test, asserting `1.0.0-from-cache`; now asserts `2.0.0-from-changelog`). 9 new in a dedicated `describe('_detectProjectVersion self-heal (#165)')` block: rewrites on version.json mismatch (asserts cache file contents + source label + recorded_at shape); rewrites on CHANGELOG mismatch (verifies live-chain priority — CHANGELOG over version.json); steady-state byte-equal preservation (no rewrite when cache matches live, filesystem-agnostic); cache preservation when no live source (git-tag-derived caches survive); read-only-directory fail-open (chmod 0o500 forces write failure, function still returns live value); creates `.tangleclaw/` if missing; non-existent path returns null without throwing; `_detectLiveVersion` returns null when no live source; `_detectLiveVersion` source labels match the reader that hit (CHANGELOG / version.json / package.json). 1880 total passing (was 1871).

## [3.16.1] - 2026-05-13


> 🛟 **Recommended bug-fix release.** Three OpenClaw / methodology-hook fixes since v3.16.0: **chunk-1 orphan-hook protection now reaches pre-#146 runtime templates** (#158) — closes the session-killing infinite Stop-hook loop that resurfaced on TC-v3 itself when chunk-1's `requires` filter silently no-op'd on pre-v3.15.0 runtime templates; **OpenClaw connection flow fixes** (#160) — surfaces real save errors, fixes Bridge Port silently defaulting to 3201 for non-ClawBridge deployments, and replaces the generic "check SSH connectivity" message with the actual SSH failure (local-bind conflict, auth failure, network error); **OpenClaw Web UI cached-WS-URL routing fix** (#162) — clicking a second OpenClaw connection's Web UI button no longer loads the first connection's dashboard via stale localStorage cache. Anyone on v3.16.0 will get the update-pill notification with a clickable link to this release page.

### Fixed

- **OpenClaw Web UI: clicking a second connection's Web UI button loaded the first connection's dashboard via cached localStorage WS URL (#162)** — When TC proxies multiple OpenClaw connections through `/openclaw-direct/<connId>/...` on the same TC origin, the iframe-loaded OpenClaw dashboard caches its derived WebSocket URL in localStorage (e.g. `wss://192.168.10.99:3102/openclaw-direct/<connId>`). Because the dashboard reads localStorage on subsequent loads instead of re-deriving from the iframe URL, the second OpenClaw's Web UI button would silently route WebSocket traffic to the *first* OpenClaw's tunnel. Symptom: clicking Claw-Node-01's Web UI button opens RentalClaw's dashboard. The iframe URL is correct; the dashboard's WebSocket connection is wrong.
  - **Scope honesty (Critic MAJOR-4)**: this fix targets the **observed manifestation** — cached values whose string content contains the proxy path `/openclaw-direct/<otherId>/`. The TC parent page can detect and remove those entries before the iframe loads. If a future OpenClaw dashboard caches its WS URL under a key whose value contains no proxy-path substring (e.g. origin-only `wss://host:port` or a structured `{host,port}` JSON), the fix as shipped does NOT clear it and the bug recurs. That broader case is acknowledged-out-of-scope here; would file as a #162-followup needing either a TC-side proxy HTML injection or a coordinated change with OpenClaw upstream.
  - **Fix in `public/openclaw-cache.js`** — new helper module `tcClearStaleOpenclawCache(connId, storage)`. Walks `localStorage`, uses `matchAll` to extract every `/openclaw-direct/<id>` reference inside each value, and deletes the entry only when EVERY reference is for a different (case-insensitive) connId than the current one. Composite values that reference both the current connection AND a stale one are PRESERVED — the dashboard's legitimate current-connection state shouldn't be lost just because the value also mentions an old id. Loaded via a new `<script src="/openclaw-cache.js">` tag in `openclaw-view.html` (positioned between `api-helper.js` and `openclaw-view.js` so the helper is defined before `openclaw-view.js`'s `init()` calls it). Uses the same UMD-ish factory pattern as `api-helper.js` — exposes on `window` for the browser and `module.exports` for tests.
  - **Wiring in `public/openclaw-view.js`** — new `setFrameSrc(frame, url)` helper is the only writer of `terminalFrame.src` in the file; it runs `tcClearStaleOpenclawCache(connId)` then sets `frame.src`. Both writer sites (`init()` initial load and the post-pairing auto-approve reload) route through this helper, so the cache-bust is symmetric (Critic MINOR-1 + MINOR-2). The iframe is same-origin with the parent (TC proxies under `/openclaw-direct/`), so localStorage is shared — clearing entries from the parent makes the dashboard JS running inside the iframe fall through to deriving its URL fresh from the iframe's own `location` (for the in-scope value shape). Call is guarded by `typeof tcClearStaleOpenclawCache === 'function'` so a missing helper script doesn't crash navigation.
  - **Policy decisions**: (a) **Composite-value safety** — `matchAll` + "hasStale && !hasCurrent" predicate: only delete when no reference inside the value matches the current connId (Critic MAJOR-2). (b) **Case-insensitive connId comparison** — UUIDs are case-insensitive per RFC 4122 §3; both extraction and comparison are normalized to lowercase so a value emitted with mixed case (uppercase / mixed canonical form) doesn't get misclassified as stale and over-deleted (Critic MAJOR-3). (c) Only entries whose VALUES reference `/openclaw-direct/<otherId>/` are touched — TC's own per-project settings under `tc_<project>_<key>` and other unrelated keys are preserved. (d) Same-connection cache is preserved (the dashboard's prior session state survives navigation away and back, including across case differences). (e) Idempotent — calling twice with the same connId does nothing the second time. (f) Storage walk goes BACKWARDS so live mutation via `removeItem` doesn't skip keys (regression-tested). (g) Tolerant of incognito/disabled storage — wrapped in try/catch, returns 0 on any thrown access. (h) Defensive against missing/empty/malformed `currentConnId` — UUID-shape validation (`/^[0-9a-fA-F-]{16,64}$/`) means an accidental empty/typo input returns 0 without touching storage (Critic MINOR-4).
  - **Critic-driven adjustments**: Independent Critic flagged 0 BLOCKERs / 3 MAJORs / 4 MINORs / 3 NITs. All MAJORs + actionable MINORs addressed in-chunk. **M2** (composite-value over/under-deletion) — original match-first-occurrence semantics could either over-delete (first match stale → whole value gone even if it also referenced current) or under-delete (first match current → preserves entry even if it referenced an old id elsewhere). Rewrote to `matchAll` + "hasStale && !hasCurrent" predicate; composite values containing the current connId are preserved. **M3** (case-policy backwards) — original strict-equality comparison would over-delete legitimate same-connection state if OpenClaw ever emitted a mixed-case UUID. UUIDs are case-insensitive per RFC 4122 §3; both extraction and comparison are now `.toLowerCase()`-normalized. NITs (comment length, count-claim verification, CHANGELOG phrasing) accepted or addressed inline.

- **OpenClaw connection flow: misleading save errors + Bridge Port silently defaulting to 3201 + tunnel "check SSH connectivity" hiding the real failure (#160)** — Three compounding UI/UX bugs that made non-ClawBridge OpenClaw connections (any deployment without a ClawBridge sidecar — e.g. a fresh Docker-based OpenClaw on a Linux host) effectively unusable until the user manually PATCHed the record via the REST API. All three closed in one PR.
  - **Bug 1 — Save toast now surfaces the real server error.** Pre-#160, the New / Edit Connection modal hardcoded `"Save failed. Name may already exist."` for any non-2xx response. The actual error (PORT_CONFLICT, BAD_REQUEST, CONFLICT, etc.) lived in the JSON response body's `error` + `code` fields but never reached the UI. `public/ui.js:saveConnection` now uses `api.lastError` + `api.lastErrorCode` (the side-channel exposed by `public/api-helper.js` since PR #84 / issue #80) to render the real message. Falls back to the previous generic message only when no error payload was returned (e.g. network failure mid-save). Backend tests pin the contract that PORT_CONFLICT responses carry both `error` (string) and `code: 'PORT_CONFLICT'` so the frontend can disambiguate.
  - **Bug 2 — Blank Bridge Port now persists as null.** Pre-#160, an empty Bridge Port field in the modal silently saved as `3201` (the ClawBridge default) for three reasons: (a) `public/ui.js:saveConnection` serialized `parseInt(value, 10) || 3201`; (b) `lib/store.js:openclawConnections.create` ran `const bridgePort = data.bridgePort || 3201;`; (c) `lib/store.js:_rowToConnection` coerced null read-back to 3201. Every non-ClawBridge connection carried 3201 in its DB row, which then drove server.js to ask `lib/tunnel.js` for a `-L 3201:127.0.0.1:3201` SSH forward — and with `-o ExitOnForwardFailure=yes` set, any local-port conflict on 3201 killed the entire tunnel. Fixed at all three layers: form submit sends `null` on empty; create accepts null verbatim; row read-back preserves null. PATCH-to-clear (sending `bridgePort: null` or `bridgePort: ''` in PUT) now actually clears. Persisted non-null values (e.g. 4501) still round-trip unchanged — only the falsy → 3201 coercion went away.
  - **Bug 3 — Tunnel failure messages now name the actual problem instead of "check SSH connectivity".** Pre-#160, `lib/tunnel.js:ensureTunnel` spawned SSH with `stdio: 'ignore'`, discarding stderr. When `tcpProbe` saw the tunnel didn't come up, the only available message was a generic "SSH tunnel spawned but port not connectable" — which `server.js` then wrapped as "Tunnel failed: …", indistinguishable from a real SSH connectivity issue. The local-bind conflict (the canonical #160 incident shape) is invisible in this message. Fixed by switching stdio to `['ignore', 'ignore', 'pipe']`, buffering stderr (4 KB cap), and routing it through a new `_formatTunnelError(stderr, extraForwards)` helper that recognizes the load-bearing failure modes — local-bind conflict (with a special hint when the conflicting port matches a secondary forward like Bridge Port), `Permission denied`/`publickey` auth failures, `Connection refused`/`No route to host`/`Could not resolve hostname` network failures — and renders them with actionable text. Unrecognized stderr falls back to surfacing the first non-empty line so the user has something to grep.
  - **Schema migration v14 → v15 (`lib/store.js`)**: `openclaw_connections.bridge_port` was `INTEGER NOT NULL DEFAULT 3201` pre-#160. SQLite can't `ALTER COLUMN` to drop NOT NULL, so the migration recreates the table preserving every existing row's data verbatim — rows that already have 3201 keep 3201 (the migration changes only the column constraint, not existing values). Wrapped with explicit `BEGIN; ... COMMIT;` SQL (the runtime DB engine is `node:sqlite`'s `DatabaseSync`, which doesn't expose a `.transaction()` wrapper). Postcondition check verifies `PRAGMA table_info` reports `notnull=0` before advancing `schema_version` — if the recreate step silently failed for any reason, the migration throws rather than marking the DB v15 with a v14 schema still in place (Critic MAJOR-3). Best-effort `ROLLBACK` in the catch block if a BEGIN landed without COMMIT. Fresh `CREATE TABLE IF NOT EXISTS` also updated for new installs.

## [3.1.4] - 2026-03-19

### Fixed

- **Event-loop blocking from git info polling caused terminal failure**: `git.getInfo()` used `execSync` with a 10-second cache TTL, spawning up to 180 git subprocesses (6 commands × 30 project directories) on nearly every `/api/projects` poll. This blocked the Node event loop for 1–4 seconds, preventing WebSocket upgrades from completing — the browser would launch but the terminal CLI never appeared. Increased cache TTL from 10s to 2 minutes and added early `rev-parse HEAD` check to skip `git log`/`git describe` on repos with no commits.
- **Git stderr flooding server log to 19MB**: `execSync` in `git.js` did not explicitly set `stdio`, allowing child process stderr (`fatal: not a git repository`, `fatal: No names found`, etc.) to leak to the parent process stderr. Combined with the launchd plist routing both stdout and stderr to the same log file, this produced ~267K lines (19MB) of noise. Added explicit `stdio: ['pipe', 'pipe', 'pipe']` to capture stderr in-process.
- **Launchd server log growing without rotation**: `StandardOutPath` and `StandardErrorPath` in the server plist pointed to `~/Library/Logs/tangleclaw-server.log` with no rotation. The app's internal logger (`~/.tangleclaw/logs/tangleclaw.log`) already handles rotation at 10MB with 3 backups. Redirected the launchd plist outputs to `/dev/null` to eliminate the redundant, unrotated log.
